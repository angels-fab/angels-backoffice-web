# 가독성 — 출력 형식 규칙

사용자가 작업 과정을 한눈에 이해할 수 있도록, 답변을 아래 시각적 규칙으로 구성한다.
GitHub Flavored Markdown의 **Alert(블록인용)** 문법을 사용한다 (지원 뷰어에서 실제 색으로 렌더링됨).

## 핵심 3요소

### 1) 🔧 Claude가 실행한 작업 → "회색 박스"
내가 직접 수행한 명령·파일 수정·설정 변경 등은 일반 블록인용(`>`, 회색 왼쪽 바)에 담는다.
무엇을 했는지 **구체적으로** 적는다 (명령어/파일경로 포함).

### 2) 💡 쉬운 의미 설명 → 회색 박스 "안의 파란 줄"
회색 박스 안에, 그 작업이 **무슨 의미인지 비전문가도 이해할 쉬운 말**을 파란 줄(`[!NOTE]`)로 덧붙인다.
전문용어 금지, 한 줄~두 줄로.

**형식 예시:**

```
> 🔧 **실행한 작업** — `cp legacy.html public/`
> 구버전 백업 파일을 소스의 public 폴더로 복사했어요.
> > [!NOTE]
> > 💡 배포할 때마다 이 백업이 자동으로 같이 올라가서, 실수로 사라지지 않게 하려는 거예요.
```

### 3) ✋ 사용자가 직접 해야 할 일 → "노란 박스"
내가 대신 못 하는, 사용자 본인이 손으로 해야 하는 일은 노란 박스(`[!WARNING]`)에 담는다.
번호로 단계를 나누고, 클릭할 주소·입력값을 정확히 적는다.

**형식 예시:**

```
> [!WARNING]
> ✋ **직접 해주세요**
> 1. https://github.com/settings/... 접속
> 2. Name 칸에 `DEPLOY_TOKEN` 입력
> 3. Add secret 클릭
```

## 적용 규칙

- "내가 한 일"과 "사용자가 할 일"이 섞인 답변에서는 **항상** 이 형식을 쓴다.
- 회색 박스(작업)와 파란 줄(의미)은 **짝**으로 간다 — 작업만 적고 의미 설명을 빼지 않는다.
- 색은 뷰어에 따라 다르게 보일 수 있으나, `[!NOTE]`/`[!WARNING]` 라벨과 🔧/💡/✋ 이모지로 구분이 유지된다.
- 표·코드블록 등 다른 서식과 함께 써도 된다. 단, 본문이 박스로만 뒤덮이지 않게 핵심에만 적용한다.
