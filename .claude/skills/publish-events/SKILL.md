# Publish Events

포털 내부 행사 신청이나 사용자가 준 행사 정보를 검증하여 `FAB_EVENTS`와 게시용 포스터에 반영한다. Google Form과 Google Drive를 사용하지 않는다.

## 작업 범위 확인

1. `src/constants/events.ts`, `src/api/events.ts`, `src/pages/Events/SubmitEventModal.tsx`, `src/pages/Events/SubmissionsAdmin.tsx`의 현재 계약을 먼저 확인한다.
2. 신청 자료는 포털의 `event_submissions`와 비공개 `event-submissions` Storage를 기준으로 한다. 접근할 수 없으면 사용자에게 행사 URL이나 신청 내용을 요청하고 추정하지 않는다.
3. `게시완료` 상태 표시는 카드를 자동 생성하지 않는다는 점을 기억한다. `FAB_EVENTS` 반영과 포스터 저장까지 완료되어야 실제 게시다.
4. 동일 ID·제목·일정의 기존 행사가 있는지 먼저 확인한다.

## 정보 검증

1. 공식 행사 사이트를 우선 확인하고, 필요한 필드가 없을 때만 검색 범위를 넓힌다.
2. 제목, 시작일·종료일, 장소, 실제 주최·주관, 링크를 교차 확인한다.
3. 신청 내용과 공식 정보가 다르면 공식 최신 정보를 사용하되 차이를 사용자에게 알린다.
4. 행사 목록 사이트를 주최·주관기관으로 추정하지 않는다. 주관기관은 가능한 한 한글 정식명으로 쓴다.

## 카드 데이터 작성

- `id`: 기존 ID와 겹치지 않는 짧은 영문 소문자 식별자를 사용한다.
- `title`: 영문 정식명 또는 약칭을 앞에 두고 한글명은 ` - ` 뒤에 둔다.
- `kind`: 현재 `eventCategory()`가 학술·교육·전시로 올바르게 분류할 수 있는 값을 사용한다.
- `start`, `end`: `YYYY-MM-DD`로 기록하고 단일 일정은 `end`를 생략할 수 있다.
- `organizer`: 검증된 실제 주최·주관기관을 기록한다.
- `summary`: 카드에서 보이는 핵심 정보만 최대 3개 작성한다.
  - 학술: 사전등록, 초록마감, Plenary 또는 Keynote
  - 교육: 신청기간, 대상, 강사
  - 전시: 관람기간, 입장방법, 규모
- 연사는 이름만 최대 4명 표시하고 더 많으면 렌더러의 `등` 처리를 사용한다.
- 날짜 문자열을 화면용으로 직접 가공하지 말고 기존 `fmtEventDate()`를 사용한다.
- 분류·상태·색상은 기존 `eventCategory()`, `eventStatus()`, 디자인 토큰을 따른다.

## 포스터 처리

1. 신청 포스터나 공식 사이트의 최신 원본을 우선 사용한다.
2. 신청 자료가 PDF면 실제 게시 전에 대표 페이지를 웹용 이미지로 변환한다.
3. 언론사 워터마크나 기자 연락처가 포함된 보도자료 이미지는 게시용 원본으로 사용하지 않는다.
4. 게시용 이미지를 `public/events/`에 저장하고 `poster: 'events/<파일명>'`으로 연결한다.
5. 카드에서 제목과 핵심 영역이 잘리지 않도록 `posterFocus`를 설정한다. 기본값으로 충분하면 생략하고 전체 표시가 필요하면 `fit: 'contain'`을 사용한다.
6. 포스터 하단과 상세 정보가 겹쳐 읽기 어려울 때만 `darkInfo: true`를 사용한다. 스크림 수치는 현재 컴포넌트 구현을 따른다.
7. 풀 포스터가 없어서 배경을 보완할 때는 원본의 주색을 유지하고 검정으로 급격히 끊기거나 세로 줄무늬가 생기지 않게 한다.

## 게시와 검증

1. `src/constants/events.ts`의 `FAB_EVENTS`에 데이터를 추가하고 게시 이미지를 연결한다.
2. 요청하지 않은 행사 페이지 구조나 공통 디자인 시스템은 함께 리팩터링하지 않는다.
3. `npm run type-check`를 실행한다.
4. `/events`에서 진행·예정 카드와 종료 목록 분류, 제목, 날짜, 포스터 크롭, 링크를 확인한다.
5. UI에 영향을 주면 PC와 모바일을 확인한다. 로그인·권한 때문에 확인하지 못한 항목은 완료했다고 추정하지 않는다.
6. 실제 카드 반영이 끝나기 전에는 신청을 게시 완료로 처리하지 않는다.

## 제외 대상

- Google Form, 폼 소유자 이메일 알림, Google Drive 업로드 절차를 사용하지 않는다.
- 행사 페이지의 캐러셀, 스와이프 임계값, 상세 패널 상태 관리를 이 스킬의 등록 작업 때문에 변경하지 않는다.
- 여러 조사 에이전트나 광범위한 검색을 자동으로 시작하지 않는다. 공식 정보가 부족할 때만 필요한 범위에서 추가 조사한다.
