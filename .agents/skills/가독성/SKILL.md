# 가독성 — 출력 형식 규칙

답변을 **"내가 한 일 / 그 의미 / 사용자가 할 일"** 세 역할로 또렷이 나눠 한눈에 읽히게 한다.

**색이나 박스에 의존하지 않는다.** 데스크톱 앱 채팅은 GFM Alert 색을 렌더링하지 않아서 `> [!NOTE]`/`[!WARNING]`는 회색 + 라벨 글자만 보여 오히려 지저분하다. 대신 **굵은 라벨 헤더 + 목록**으로 구분한다. (이모지는 글리프라 색과 무관하게 어디서나 보인다.)

## 핵심 3요소 — 굵은 라벨로 구분

각 역할을 아래 라벨로 시작하고, 내용은 짧은 목록으로 적는다.

**🔧 한 일** — 내가 직접 수행한 명령·파일 수정·설정 변경. 구체적으로(명령어/파일경로 포함), 불릿으로.

**💡 무슨 의미냐면** — 그 작업이 무슨 뜻인지 비전문가도 이해할 쉬운 말. 전문용어 금지, 1~2줄. "한 일"과 짝으로 가며 빼지 않는다.

**✋ 직접 하실 일** — 내가 대신 못 하는, 사용자 본인이 손으로 할 일. 번호 목록으로, 클릭할 주소·입력값을 정확히.

## 형식 예시

```
**🔧 한 일**
- `index.css`에서 캘린더 박스 스타일 수정
- 타입체크·빌드 통과

**💡 무슨 의미냐면**
- 일정 칸을 클릭하면 수정/삭제 창이 열리게 했어요.

**✋ 직접 하실 일**
1. http://localhost:3600 접속
2. 일정 추가 → 이름·비밀번호 입력 → 추가
3. 구글 캘린더에 생겼는지 확인
```

## 적용 규칙

- "내가 한 일"과 "사용자가 할 일"이 섞인 답변에서는 **항상** 이 형식을 쓴다.
- "한 일"과 "의미"는 **짝** — 작업만 적고 의미 설명을 빼지 않는다.
- 라벨 사이는 빈 줄로 띄워 단락이 뭉치지 않게 한다.
- **GFM Alert(`> [!NOTE]`/`[!WARNING]`)와 색 박스 위젯은 쓰지 않는다.** 일반 텍스트로만.
- 표·코드블록과 함께 써도 되며, 짧은 답변은 라벨 한두 개만 써도 된다. 본문 전체를 라벨로 도배하지 않는다.
- 단순 대화·한 줄 답변엔 억지로 적용하지 않는다.
